# 远程安装应用性能优化设计

## 目标
在不做大规模架构重写的前提下，优先提升两个核心体感指标：
- 应用启动后主界面尽快可交互
- 主机切换、刷新时的应用检测尽快给出可用结果

本次设计不追求一次性把所有内部逻辑都“跑得更快”，而是优先重构首屏路径与检测触发策略，让用户更早看到界面、避免重复检测，并减少不必要的 SSH 往返。

## 背景与现状
当前已确认的关键现状如下：

### 1. `App.xaml.cs` 不是主要瓶颈
`RemoteInstaller/App.xaml.cs:15-42` 的启动逻辑只包含：
- 设置渲染模式
- 注册 UTF-8 编码

这一段本身较轻，不应作为本轮性能优化重点。

### 2. `MainViewModel` 构造阶段承担了大量首屏职责
`RemoteInstaller/ViewModels/MainViewModel.cs:1553-1560` 当前在构造函数中串行执行：
- `LoadSettings()`
- `LoadHosts()`
- `InitializeSampleData()`
- `ApplyServerFilter()`
- `ApplyAppFilter()`
- `ApplyTaskFilter()`
- `LoadCustomApps()`

这意味着主窗口可见前，ViewModel 已经做了多段初始化工作。即使其中单项不重，叠加后也会影响首屏可交互时间。

### 3. 主机切换会自动触发全量检测
`RemoteInstaller/ViewModels/MainViewModel.cs:289-296` 中，`SelectedHost` 变化后会立即执行：
- `_ = CheckAllAppsStatusAsync(SelectedHost);`

此外，`RemoteInstaller/ViewModels/MainViewModel.cs:657-665` 在自定义应用部署对话框关闭后也会直接触发：
- `_ = CheckAllAppsStatusAsync(SelectedHost);`

这使“切换主机”“关闭对话框”等交互天然附带远程检测成本。

### 4. 检测逻辑已做局部优化，但触发层仍然粗放
`RemoteInstaller/ViewModels/MainViewModel.cs:1894-2024` 中当前检测链路已经有两个正确方向：
- 单主机内对应用状态检查使用 `SemaphoreSlim(3)` 控制并发
- 所有结果汇总后一次性回写 UI，避免高频 Dispatcher 更新

但仍存在以下问题：
- 选中主机后立即检测，缺少去抖
- 同一主机短时间内可能重复触发多次全量检测
- 没有主机级最近结果缓存
- 单应用检查仍可能产生较多 SSH 往返
- 用户快速切换主机时，检测成本会不断叠加

## 设计原则
1. **体感优先**：优先缩短“看见界面并能操作”的时间，而不是要求所有后台状态都提前准备完。
2. **最小侵入**：尽量在现有 `MainViewModel + InstallerService + SSH` 结构上收敛问题，不做大规模重构。
3. **先控触发，再压耗时**：先解决重复检测、错误触发时机、无缓存，再优化单次检测内部实现。
4. **保留现有优点**：继续保留“检测结果批量回写 UI”的模式，不回退成逐项频繁刷新。
5. **分层加载**：把首屏必需数据与后台可延迟数据严格区分。
6. **主机级协同**：性能策略应以“主机”为单位管理缓存、去抖、去重和取消，而不是把这些逻辑散落在多个调用点。

## 方案比较

### 方案 A：激进懒加载
核心做法：
- 启动时只加载最基础 UI 数据
- 不做任何自动远程检测
- 用户显式操作后再开始检测

优点：
- 冷启动体感提升最大
- 实现目标明确

缺点：
- 行为变化较大
- 用户首次查看主机时需要等待更明显
- 与当前“选中主机自动刷新”习惯不完全一致

### 方案 B：仅优化检测内部实现
核心做法：
- 保持现有启动流程与交互触发方式
- 重点优化 `CheckAllAppsStatusAsync(...)` 的并发和命令组织

优点：
- 行为变化小
- 对检测耗时改善直接

缺点：
- 冷启动收益有限
- 仍会存在“无意义重复触发”的问题

### 方案 C：分层优化（推荐）
核心做法：
- 启动阶段减负，延后非关键工作
- 检测阶段增加去抖、取消、缓存、去重
- 后续再压缩 SSH 往返次数

优点：
- 同时改善启动与检测体感
- 风险较低，适配现有代码结构
- 可分阶段落地，便于持续验证

缺点：
- 需要在多个调用点统一收敛触发规则
- 收益来自多处叠加，而非单点爆改

本次设计采用 **方案 C**。

## 设计总览
本次性能优化分为三层：

### 第一层：首屏路径减负
窗口显示所必需的数据继续同步加载；不影响首屏展示的内容改为后台补齐。

### 第二层：检测触发收敛
所有主机状态检测入口统一走主机级调度，增加去抖、取消、缓存、去重，避免每个调用点都直接 `_ = CheckAllAppsStatusAsync(...)`。

### 第三层：检测执行压缩
在不破坏现有检测结果结构的前提下，逐步把多次 SSH 小调用合并成更少的远程检查批次，降低网络往返成本。

## 启动路径设计

### 1. 首屏同步层
以下内容保留在窗口首次可用前完成：
- 主题设置与仓库地址等基础设置读取
- 主机列表基础信息加载
- 应用市场静态定义加载
- 初始筛选集合构建

对应现有方法包括：
- `LoadSettings()`
- `LoadHosts()`
- `InitializeSampleData()` 中与应用定义加载直接相关的部分
- `ApplyServerFilter()` / `ApplyAppFilter()` / `ApplyTaskFilter()`

这些内容的共同特征是：
- 直接影响首屏结构是否完整
- 不依赖远程 SSH
- 用户很容易立即感知其缺失

### 2. 首屏后后台层
以下内容应改为窗口显示后异步补齐：
- 自定义应用加载
- 首次状态检测
- 非关键日志补充
- 任何远程 SSH 行为

其中 `LoadCustomApps()` 当前位于构造函数末尾，可优先考虑挪到窗口可见后异步执行。这样能先保证主界面可交互，再补齐扩展数据。

### 3. 首次状态展示策略
启动后默认不要求所有主机立刻完成检测。界面允许先显示：
- 上次缓存结果
- 未知状态
- 正在刷新中

重点是避免“为了得到完整状态而阻塞主界面显示”。

## 检测触发设计

### 1. 统一检测入口
当前多个位置会直接调用 `CheckAllAppsStatusAsync(...)`，例如：
- `MainViewModel.cs:295`
- `MainViewModel.cs:664`
- 刷新、连接测试、批量检测等其它路径

本次设计要求新增一个统一的主机检测调度入口，由它负责：
- 接收检测请求
- 判断是否需要立即执行
- 判断是否已有同主机检测在进行
- 判断是否可以复用缓存
- 管理取消和去抖

`CheckAllAppsStatusAsync(...)` 保留为底层实际执行方法，不再由 UI 事件和各业务入口直接散落调用。

### 2. 主机切换去抖
当 `SelectedHost` 变化时，不再立即全量检测，而是：
- 先记录当前选中主机
- 启动一个短暂去抖窗口
- 若用户在去抖窗口内再次切换主机，则取消前一个待执行请求
- 只有主机选择稳定后，才真正开始检测

这样可以显著降低快速切换列表时的无效远程调用。

### 3. 同主机检测去重
对于同一台主机：
- 如果已有检测在执行，则不再重复发起第二轮全量检测
- 新请求可选择复用现有任务，或取消旧任务后只保留最新请求

推荐策略：
- “选中主机变化”类请求：取消旧请求，保留最新请求
- “手动刷新”类请求：若已有同主机检测在执行，则直接复用当前检测任务，并在 UI 上体现“正在刷新”

### 4. 关闭对话框后的刷新收敛
如 `RefreshCustomAppsStatusAfterDialog()` 当前会直接触发全量检测。该类场景应改为：
- 进入统一调度入口
- 若最近刚完成同主机检测且缓存仍新鲜，可直接复用缓存
- 若检测正在执行，则不再额外重复触发

这样可以避免“刚部署结束就立刻重复打 SSH”造成的额外等待。

## 检测缓存设计

### 1. 缓存粒度
缓存按“主机”维度维护，而不是按单个应用散落缓存。每条缓存至少包含：
- 主机标识
- 最近一次检测时间
- 内置应用状态集合
- 自定义应用状态集合

### 2. 缓存使用规则
当用户切换到某台主机时：
- 若缓存仍在新鲜窗口内，界面先显示缓存结果
- 同时可根据策略决定是否后台静默刷新
- 若缓存已过期，则显示缓存结果并标记“刷新中”，或直接进入新检测

这样可把“切回刚看过的主机”从重新等待一次，变成几乎立即可见。

### 3. 缓存失效时机
以下情况应使主机缓存失效或标记为脏：
- 对该主机发起安装/卸载操作后
- 自定义应用部署变更后
- 手动刷新时
- 用户显式要求重新检测时

缓存不是长期真相，而是为了提升体感的近期快照。

## 检测执行设计

### 1. 保留现有批量回写 UI 模式
当前 `CheckAllAppsStatusAsync(...)` 先收集 `results/customResults`，再统一回写 UI，这一点应保留。

原因：
- 避免高频 Dispatcher 更新
- 避免列表卡片频繁抖动
- 更符合桌面端批量状态刷新的展示体验

### 2. 减少 SSH 往返次数
当前单主机内虽然有并发限制，但每个应用仍可能单独执行状态检查。后续应优先把以下内容合并：
- Linux 服务运行状态检查
- 安装目录/文件存在性检查
- 常见版本探测

目标不是把并发无限调大，而是减少 round-trip 数量。

### 3. 自定义应用纳入统一调度
自定义应用当前在 `CheckSingleCustomAppStatusAsync(...)` 中单独处理。后续不要求立即重写逻辑，但要纳入统一主机检测调度体系，与内置应用共享：
- 同一轮检测的生命周期
- 缓存写入
- 取消策略
- UI 完成回写时机

### 4. 结果无变化不重复刷新
当新检测结果与当前 UI 状态一致时，不必无条件刷新所有属性。可在统一回写前增加比较：
- 安装状态是否变化
- 运行状态是否变化
- 版本/状态文案是否变化

只有发生变化的项才更新对应 ViewModel 字段，避免不必要的绑定更新和视觉扰动。

## 职责边界设计

### 当前问题
`MainViewModel` 当前同时承担：
- 启动初始化
- 交互触发
- 检测调度
- 状态缓存思考
- UI 更新

这会让性能策略散落在多个方法中，后续继续优化会越来越难。

### 推荐边界
本次不做大重构，但应增加一个很薄的“检测协调层”或等价职责单元。其职责包括：
- 主机级去抖
- 同主机检测去重
- 检测任务取消
- 最近结果缓存
- 是否需要真正执行检测的判断

职责划分建议：

#### `MainViewModel`
负责：
- 选择主机
- 触发刷新
- 接收结果并更新界面状态
- 展示“刷新中 / 使用缓存 / 已完成”反馈

#### 检测协调层
负责：
- 管理主机检测请求生命周期
- 管理缓存和失效
- 决定复用、取消还是重新发起检测

#### `InstallerService`
继续负责：
- 单个应用的状态检查
- 后续可逐步承接批量检查接口

这样能在不大拆现有结构的前提下，把“什么时候查”和“怎么查”分开。

## UI/交互行为约束
1. 不为了性能优化改动主界面整体结构。
2. 不移除用户现有的刷新能力。
3. 允许主机切换后先看到缓存或未知状态，再渐进刷新。
4. 对正在刷新中的主机，应给出明确但轻量的反馈，不让界面看起来像卡死。
5. 性能优化不应改变安装、卸载、日志等现有业务语义。

## 分阶段落地建议

### 第一阶段：先拿最大体感收益
- 启动阶段移除自动远程检测
- `SelectedHost` 检测改为去抖触发
- 同主机检测加去重/取消
- 增加主机级最近结果缓存

这一阶段应优先完成，因为收益最大、风险最可控。

### 第二阶段：压缩单次检测耗时
- 合并 Linux 常见状态检查命令
- 减少逐应用独立 SSH 调用
- 把自定义应用检测纳入统一批次调度

这一阶段主要优化真实检测耗时。

### 第三阶段：优化速度感知
- 首屏优先展示缓存/未知态
- 局部更新变化项
- 明确“刷新中”状态展示

这一阶段主要优化用户的感知速度与稳定性。

## 验证方案
按当前项目偏好，本次不跑测试，以构建和定向行为验证为主。

### 1. 构建验证
执行：
- `dotnet build RemoteInstaller/RemoteInstaller.csproj`

期望：
- 构建通过
- 无新增编译错误

### 2. 启动体验验证
检查：
- 打开应用后，主窗口更早进入可交互状态
- 首屏不因远程检测阻塞
- 主机列表与应用列表基础结构能立即展示

### 3. 主机切换验证
检查：
- 快速连续切换主机时，不再每次都立即发起全量检测
- 稳定停留在某台主机后才开始实际检测
- 旧主机的待执行检测能被取消或忽略

### 4. 缓存验证
检查：
- 切回最近刚查看过的主机时，能优先看到最近结果
- 安装/卸载/部署后，该主机缓存会失效或被标脏
- 手动刷新可强制重新检测

### 5. 检测行为验证
检查：
- 同一主机短时间内不会并发堆积多轮全量检测
- 检测完成后仍是批量更新 UI，而不是逐项频繁跳动
- 如果结果没有变化，不会发生明显的重复刷新闪动

## 非目标
本次不做以下事项：
- 不重写整个 MVVM 架构
- 不替换 SSH/InstallerService 基础实现
- 不把所有应用状态检查一次性全部改成统一脚本协议
- 不改动任务进度、安装流程、MariaDB 安装逻辑等无关模块
- 不引入复杂性能监控框架或新的基础设施组件

## 结论
本设计采用“首屏减负 + 检测调度收敛 + 执行压缩”的分层路线：
- 启动阶段只保留首屏必需工作，非关键内容延后
- 所有主机检测入口统一调度，增加去抖、取消、缓存、去重
- 保留现有批量回写 UI 的优点，并逐步减少 SSH 往返次数

这样可以在不做粗暴大改的前提下，同时改善启动速度与检测速度，并把后续性能优化建立在更稳定的职责边界之上。
